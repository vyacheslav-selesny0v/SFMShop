1. Анализ требований проекта

Связи: 
Пользователь связан с заказом, заказ связан с конкретным товаром.

Целостность: 
Нельзя создать заказ для несуществующего пользователя. Нельзя добавить товар в заказ, если товара нет.

Сложные запросы:
Возможность быстро выгрузить историю покупок конкретного человека со всеми деталями.

Стабильность схемы: 
Поля в таблице товаров (название, цена, описание) обычно фиксированы.

2. Сравнение PostgreSQL и MongoDB

PostgreSQL (SQL): Отлично справляется со связями (JOIN) и гарантирует, что данные не потеряются и не продублируются ошибочно (ACID-соответствие).

MongoDB (NoSQL): Хранит данные в виде гибких документов (как JSON). Она быстрее работает, когда данных очень много и их структура постоянно меняется, но хуже справляется со сложными связями между разными сущностями.

3. Почему выбран PostgreSQL

PostgreSQL выбран, потому что данные SFMShop строго структурированы, требуют строгой проверки связей и выполнения сложных транзакций при оформлении заказов.

4. Когда MongoDB был бы лучше

Для логов (запись действий пользователей в реальном времени).
Для каталога товаров с очень разными характеристиками (например, если у одного товара 2 параметра, а у другого 10)
